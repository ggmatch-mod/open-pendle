## Pendle V2 permissionless market ("Community Pool") creation on Arbitrum — contract-level mechanics and Pendle's own UI/flow (verified July 2026)

**Summary:** Pendle V2 market creation is fully permissionless at the contract level. The current stack on Arbitrum (deployments/42161-core.json keys "yieldContractFactoryV6" and "marketFactoryV6") is PendleYieldContractFactoryUpg (VERSION()=6) at 0xBA814Bf6E27A6d6baE4a8aC65c8Bc3d8e9B0aaCF and PendleMarketFactoryV7Upg (VERSION()=7) at 0x49F2f7002669E0e4425Fa0203975625Ab4af3143 — note the deployment-key vs on-chain-VERSION naming mismatch. createYieldContract(address SY, uint32 expiry, bool doCacheIndexSameBlock) deploys a PT+YT pair for any SY with expiry > now and expiry % expiryDivisor == 0 (live expiryDivisor = 86400, i.e. midnight UTC; Pendle's own tooling additionally enforces Thursday 00:00 UTC via expiry % 604800 == 0); one PT per (SY,expiry) per factory. createNewMarket(address PT, int256 scalarRoot, int256 initialAnchor, uint80 lnFeeRateRoot) deploys the AMM; PT must come from the factory's paired yield-contract factory, initialAnchor >= 1e18, lnFeeRateRoot <= ln(1.05); multiple markets per PT with different params are allowed. Pendle's official bootstrap path is PendleCommonPoolDeployHelperV2 ("commonDeploy", 0x2Ed473F528E5B320f850d17ADfe0e558f0298aA9, verified on-chain to point at the V6 factories, the CommonSYFactory and Router 0x8888...F946), which in one tx optionally deploys a template SY, creates PT/YT, computes scalarRoot/initialAnchor/lnFeeRateRoot from user-friendly PoolConfig {expiry, rateMin, rateMax, desiredImpliedRate, fee} via MarketDeployLib, then seeds liquidity (mint SY -> mint PT+YT on a computed proportion -> addLiquidityDualSyAndPt; LP+YT returned to caller). Pendle's "community pool" UX is NOT an in-app wizard: it is the listing.pendle.finance portal (checklists + on-chain risk-acknowledgment signature) plus self-deployment via the Pendle-Common-Pool-Deploy hardhat repo, after which Pendle's team does backend indexing and UI whitelisting — the app only shows curated pools. Protocol fees (5% of YT interest, 5% of rewards, 80% of swap fees) are hardcoded to flow to Pendle's treasury 0xCbcb48e22622a3778b6F14C2f5d258Ba026b05e6 at contract level regardless of frontend, so OpenPendle satisfies the fee requirement automatically by using the canonical factories/router.

**Key facts:**
- createYieldContract exact signature (PendleYieldContractFactoryUpg, VERSION=6): function createYieldContract(address SY, uint32 expiry, bool doCacheIndexSameBlock) external returns (address PT, address YT). No access control — comment in source: 'Anyone can create a yield contract'. Reverts YCFactoryInvalidExpiry if expiry is in the past OR expiry % expiryDivisor != 0; reverts YCFactoryYieldContractExisted if getPT[SY][expiry] != address(0).
- Expiry constraints: live expiryDivisor on Arbitrum V6 factory = 86400 (eth_call verified) so expiries must be at 00:00:00 UTC; expiry is uint32 (max ~year 2106, intentional 'fat finger' guard); no on-chain minimum duration beyond 'not in the past'; expiryDivisor is owner-settable (governance). Pendle's official deploy script additionally requires expiry % 604800 == 0 ('Should be Thursday' 00:00 GMT) — a convention, not an on-chain rule.
- One PT/YT pair per (SY, expiry) PER FACTORY, but five factory generations coexist on Arbitrum (yieldContractFactory 0x28dE02Ac3c3F5ef427e55c321F73fDc7F192e8E4, V3 0xEb38531db128EcA928aea1B1CE9E5609B15ba146, V4 0xc7F8F9F1DdE1104664b6fC8F33E49b169C12F41E, V5 0xFF29e023910FB9bfc86729c1050AF193A45a0C0c, V6 0xBA814Bf6E27A6d6baE4a8aC65c8Bc3d8e9B0aaCF), so duplicate PTs for the same SY+expiry ARE possible across factories. OpenPendle must handle this when resolving/validating markets.
- createYieldContract deploys PT via Create2 (salt = bytes32(block.chainid)) with name 'PT <syCoreName> <date>' / symbol 'PT-<syCoreSymbol>-<date>' and same decimals as SY.assetInfo() assetDecimals; YT deployed via BaseSplitCodeFactory._create2; then PT.initialize(YT). It reads SY.assetInfo(), SY.name(), SY.symbol() — any contract implementing IStandardizedYield works; there is NO SY allowlist.
- doCacheIndexSameBlock: caches the PY index so it updates at most once per block. Pendle's helper sets it to (block.chainid == 1), i.e. true on Ethereum mainnet, false on Arbitrum.
- Active market factory on Arbitrum: deployments key marketFactoryV6 = 0x49F2f7002669E0e4425Fa0203975625Ab4af3143, contract class PendleMarketFactoryV7Upg, on-chain VERSION() = 7 (verified via eth_call). Its immutable yieldContractFactory = 0xBA814Bf6...aaCF (V6 YCF) — createNewMarket requires IPYieldContractFactory(yieldContractFactory).isPT(PT), so the PT MUST have been created on the paired V6 yield-contract factory. Older marketFactoryV5 0xd29e76c6F15ada0150D10A1D3f45aCCD2098283B pairs with YCF V5.
- createNewMarket exact signature: function createNewMarket(address PT, int256 scalarRoot, int256 initialAnchor, uint80 lnFeeRateRoot) external returns (address market). Permissionless ('Anyone is allowed to create a market on their own'). Checks: isPT on paired YCF; !PT.isExpired(); lnFeeRateRoot <= maxLnFeeRateRoot = ln(1.05) = 48790164169432003 (~4.879e16, eth_call verified); markets[PT][scalarRoot][initialAnchor][lnFeeRateRoot] must be empty (so multiple markets per PT are fine if any param differs); initialAnchor >= minInitialAnchor = 1e18. Emits CreateNewMarket(market, PT, scalarRoot, initialAnchor, lnFeeRateRoot); event topic0 = 0x7bfe41f13edfa2a1d07e900c4ad59babb2f2ae76c66f75233f93fdbe613cb877. Factory exposes isValidMarket(address) per-factory.
- AMM economics (MarketMathCore.sol): exchangeRate(p) = ln(p/(1-p))/rateScalar + rateAnchor where p = PT proportion; rateScalar = scalarRoot * (365 days) / timeToExpiry (liquidity concentrates as expiry nears); rateAnchor is recomputed every trade from lastLnImpliedRate to preserve rate continuity (initialAnchor only sets the launch state via setInitialLnImpliedRate at first liquidity). Hard limits: PT proportion after trade <= MAX_MARKET_PROPORTION = 0.96e18 (96%), exchangeRate >= 1 (no negative implied rates), MINIMUM_LIQUIDITY = 1000 LP wei locked on first mint (first LP gets sqrt(syDesired*ptDesired) - 1000).
- How Pendle derives factory params from user-friendly inputs (MarketDeployLib.sol, exact math): given PoolConfig {uint32 expiry; uint256 rateMin; uint256 rateMax; uint256 desiredImpliedRate; uint256 fee} (all rates 1e18-scaled APYs): T = yearsToExpiry; initialAnchor = ((1+rateMin)^T + (1+rateMax)^T) / 2; scalarRoot = ln(9) * T / ((1+rateMax)^T - initialAnchor) with LN_9 = 2197224577336219648. This maps the implied-APY band [rateMin, rateMax] onto PT-proportion band [0.1, 0.9] at launch. lnFeeRateRoot = ln(1 + fee) (calcFee). Initial PT proportion for seeding: p = logitP/(1+logitP) where logitP = exp((e^(ln(1+desiredImpliedRate)*t/1y) - initialAnchor) * rateScalar) (calcInitialProportion).
- Fee semantics: 'fee' is in implied-rate terms (e.g. 0.008e18 = 0.8% rate fee); per-swap fee multiplier = e^(lnFeeRateRoot * timeToExpiry / 365d) applied against the trader (calcTrade divides/multiplies preFeeExchangeRate by it), so absolute fees decay toward expiry. Cap: fee <= 5% rate terms (ln(1.05)). Pendle's official deploy script uses the heuristic fee = rateMax / 25 (e.g. 20% max APY -> 0.8% fee) — no other 'fee tier presets' were found. reserveFeePercent = 80 on Arbitrum (eth_call verified): 80% of swap fees go to Pendle treasury, 20% to LPs. Factory owner can setOverriddenFee(router, market, fee) only DOWNWARD per (router, market).
- Protocol fees to Pendle treasury are contract-enforced regardless of frontend: V5/V6 YCF interestFeeRate = rewardFeeRate = 5e16 (5% of YT-accrued interest and 5% of external rewards, caps 20%, eth_call verified; the 2023-era factory was 3%), plus 80% of swap fees. Treasury = 0xCbcb48e22622a3778b6F14C2f5d258Ba026b05e6 on all active factories (eth_call verified). OpenPendle needs zero extra work to keep fees flowing — it just must use canonical factories/markets.
- Official one-tx bootstrap path: PendleCommonPoolDeployHelperV2 ('commonDeploy') at 0x2Ed473F528E5B320f850d17ADfe0e558f0298aA9 (used by Pendle's own Pendle-Common-Pool-Deploy repo; eth_call verified wiring: marketFactory=V6 0x49F2..., yieldContractFactory=V6 0xBA81..., router=0x888888888889758F76e7103c6CbF23ABbF58F946, syFactory=0x466CeD3b33045Ea986B2f306C8D0aA8067961CF8). Key permissionless entrypoint OpenPendle can reuse for ANY existing SY: deploy5115MarketAndSeedLiquidity(address SY, PoolConfig config, address tokenToSeedLiquidity, uint256 amountToSeed) public payable returns (PoolDeploymentAddrs {SY, PT, YT, market}). Flow: calcParams -> createYieldContract if getPT empty -> createNewMarket -> transferIn seed token -> router.mintSyFromToken (if token != SY) -> mintPyFromSy on amountSY*initialProportion -> addLiquidityDualSyAndPt(msg.sender, ...) with remaining SY + all minted PT; LP goes to caller and the minted YT is transferred back to caller. Emits MarketDeployment(addrs, params).
- One-click SY deployment (adjacent, verified): PendleCommonSYFactory at 0x466CeD3b33045Ea986B2f306C8D0aA8067961CF8 — deploySY(bytes32 id, bytes constructorParams, address syOwner) and deployUpgradableSY(id, constructorParams, initData, syOwner) are permissionless but only for owner-registered creation-code ids. Helper wraps them: deployERC4626Market, deployERC4626NotRedeemableMarket, deployERC20Market (constructorParams = abi.encode(string name 'SY ...', string symbol 'SY-...', address yieldToken)) plus WithAdapter variants and deployCommonMarketById. IDs: keccak256('PendleERC4626SYV2'), 'PendleERC4626NotRedeemableToAssetSYV2', 'PendleERC20SY', 'PendleERC4626WithAdapterSY', 'PendleERC4626NoRedeemWithAdapterSY', 'PendleERC20WithAdapterSY', 'PendleERC4626NoRedeemNoDepositUpgSY'. Upgradeable SYs are TransparentUpgradeableProxy under Pendle proxyAdmin 0xA28c08f165116587D4F3E708743B4dEe155c5E64 (eth_call verified); Pendle's script passes syOwner = governanceProxy 0x2aD631F72fB16d91c4953A7f4260A97C2fE2f31e. Factory nonce on Arbitrum = 12 (12 SYs deployed through it so far).
- Pendle's community-pool 'UI' today (docs Integration/CommunityListing, verified July 2026): NOT an in-app creation wizard. Flow = (1) listing.pendle.finance portal: auth + mandatory checklists (Asset Launching, Security checklist for ERC-4626 / non-4626, Token Pricing & Metadata); (2) build custom SY if asset isn't a plain ERC20/4626 (requirements for listing: fully transferable ERC-20 underlying, no locking/vesting on mint, up-only yield, audit for custom SYs); (3) self-deploy via the Pendle-Common-Pool-Deploy hardhat repo (fill src/deploy.ts config: name/symbol/yieldToken/type, expiry, rateMin, rateMax, desiredImpliedRate, tokenToSeed, amountToSeed); (4) deployer signs an on-chain risk-acknowledgment message; (5) seed liquidity (Pendle recommends deploying with <$10 seed, then topping up via their UI once whitelisted); (6) Pendle core team does backend setup, UI whitelisting, and optional PENDLE incentive/gauge enablement; (7) partner gets a Pool Admin Portal for metadata. Docs explicitly: on-chain creation is open to all, but 'visibility of these markets on the official Pendle UI is curated through a review process'.
- Bootstrapping pitfalls (verified in code/docs): (a) 'The yield range is immutable once the pool is deployed' — if implied APY moves outside [rateMin, rateMax] the pool goes out-of-range (96% proportion cap / exchangeRate>=1 floor) and a new market must be deployed; (b) desiredImpliedRate must lie strictly within (rateMin, rateMax) (script validates; otherwise seeding hits extreme proportions/reverts); (c) trades revert if totalPt == 0 or totalAsset == 0 (MarketZeroTotalPtOrTotalAsset) — must seed BOTH sides via dual liquidity, which the helper guarantees; (d) initialAnchor must be >= 1e18 (no negative-rate markets); (e) identical (PT, scalarRoot, initialAnchor, lnFeeRateRoot) tuple cannot be redeployed on the same factory; (f) seeding returns YT + LP to the seeder — the seeder ends up long-yield unless they sell the YT.
- Gaps for OpenPendle to fill (vs Pendle's current offering): no load-market-by-pasted-address in Pendle's app (only backend-indexed, team-whitelisted pools are visible/tradeable there); no in-browser one-click deploy (hardhat scripts + portal instead); markets created outside the listing process trade fine on-chain via Router 0x8888...F946 (router takes the market address as an explicit parameter) but get no Pendle UI, no PENDLE gauge incentives, and no backend/API data; validating a pasted address requires checking isValidMarket() across all factory generations and reading market.readTokens() for SY/PT/YT. An ExpiredLpPtRedeemer helper exists on Arbitrum at 0x23567b248cd64479384d2e0cbe83522afb8dd446 for expired-market redemptions.

## SY (EIP-5115) adapters and permissionless SY creation for OpenPendle (Arbitrum, verified July 2026)

**Summary:** Pendle already ships everything OpenPendle needs for permissionless SY creation on Arbitrum: an on-chain, permissionless-to-call SY factory (PendleCommonSYFactory at 0x466CeD3b...61CF8) that deploys Pendle-audited SY templates (plain ERC-20, ERC-4626, ERC-4626-not-redeemable, plus adapter/upgradeable variants) from governance-registered creation code, and a one-transaction helper (PendleCommonPoolDeployHelperV2 at 0x2Ed473F5...8aA9) that deploys SY + PT/YT + market and seeds liquidity. No SY whitelisting exists at the protocol level: PendleYieldContractFactoryUpg.createYieldContract() accepts ANY address implementing IStandardizedYield ("Anyone can create a yield contract"), and PendleMarketFactoryV7Upg.createNewMarket() is likewise permissionless; Pendle's curation applies only to its own app UI/backend (community listing portal), which OpenPendle bypasses. Protocol fees flow to Pendle's treasury automatically (5% interest/reward fee on YT, 80% reserve of market swap fees on Arbitrum, live-read July 2026), so OpenPendle needs no fee plumbing. OpenPendle therefore does NOT need to ship its own SY factory for plain ERC-20/ERC-4626 assets; if it did, EIP-1167 minimal-proxy clones are incompatible with Pendle's current templates because yieldToken/asset/decimals are constructor-set immutables (even in the Upg bases) — the prior art is Pendle's own split-creation-code CREATE2 factory, which preserves audited bytecode per deployment. Rebasing tokens are NOT supported by the common templates and need bespoke SYs.

**Key facts:**
- IStandardizedYield (EIP-5115) essentials, verified from pendle-core-v2-public/contracts/interfaces/IStandardizedYield.sol: deposit(address receiver, address tokenIn, uint256 amountTokenToDeposit, uint256 minSharesOut) payable returns (uint256 amountSharesOut); redeem(address receiver, uint256 amountSharesToRedeem, address tokenOut, uint256 minTokenOut, bool burnFromInternalBalance) returns (uint256 amountTokenOut); exchangeRate() view (invariant: exchangeRate * syBalance / 1e18 = asset balance); getTokensIn()/getTokensOut()/isValidTokenIn(address)/isValidTokenOut(address); previewDeposit(tokenIn, amount) view; previewRedeem(tokenOut, shares) view; claimRewards(address user) returns uint256[] (non-view); accruedRewards(address user) view; rewardIndexesCurrent() (non-view); rewardIndexesStored() view; getRewardTokens() view; yieldToken() view; assetInfo() view returns (AssetType{TOKEN=0,LIQUIDITY=1}, address assetAddress, uint8 assetDecimals). SY is itself an ERC-20 with permit; deposit/redeem are nonReentrant.
- SYBase (and SYBaseUpg) are Ownable + Pausable: owner can pause(), which blocks _beforeTokenTransfer and therefore ALL SY mints/burns/transfers (bricking deposits, redemptions and market trading). The SY factory force-transfers ownership to the caller-supplied syOwner; Pendle's own deploy script passes Pendle's governanceProxy (0x2aD631F72fB16d91c4953A7f4260A97C2fE2f31e on Arbitrum) as syOwner. OpenPendle should default syOwner to Pendle governance to match Pendle's trust profile, or clearly warn users if the deployer keeps ownership (pause = grief/freeze vector; setAdapter on adapter SYs = potential theft vector).
- Accounting modes in the common templates (verified source, Pendle-SY-Public repo): PendleERC20SY — 1 share = 1 token, exchangeRate() = 1e18, tokensIn/out = [erc20]; ONLY valid for non-rebasing, non-fee-on-transfer ERC-20s (a rebasing token breaks its 1:1 accounting). ## PendleRouter (V4) integration for OpenPendle without Pendle's hosted backend: function signatures, ApproxParams, client-side quoting, hosted API/SDK status, zap architecture, Arbitrum addresses, limit orders

**Summary:** Pendle's current router is the "Router V4" diamond-style proxy at 0x888888888889758F76e7103c6CbF23ABbF58F946 (same vanity address on all chains, confirmed on Arbitrum via deployments/42161-core.json in pendle-core-v2-public). It exposes the V3 action facets (ActionSwapPTV3, ActionSwapYTV3, ActionMiscV3, ActionAddRemoveLiqV3, ActionSimple, ActionCallbackV3, ActionStorageV4). Every operation OpenPendle needs is directly callable on-chain with no Pendle backend: swapExactTokenForPt/swapExactPtForToken (+Sy variants), swapExactTokenForYt/swapExactYtForToken (+Sy variants), mintPyFromToken/Sy + redeemPyToToken/Sy, mintSyFromToken/redeemSyToToken, dual and single-sided add/remove liquidity, and exitPreExp*/exitPostExp* for matured markets. Functions taking ApproxParams run an on-chain binary search; the documented safe default is ApproxParams(guessMin=0, guessMax=type(uint256).max, guessOffchain=0, maxIteration=256, eps=1e14) — also codified as createDefaultApproxParams() in IPAllActionTypeV3.sol — and supplying a good guessOffchain skips the search entirely to save gas. Client-side quoting for arbitrary markets is fully feasible without any backend: RouterStatic (0xAdB09F65bd90d19e3148D9ccb693F3161C6DB3E8 on Arbitrum) provides ~27 static quote functions (swapExactTokenForPtStatic, addLiquiditySingleTokenStatic, etc., returning netOut, fees, priceImpact, post-trade exchange rate) that work for any market address; alternatively viem simulateContract/eth_call against the real router functions gives exact execution results (needs balances/approvals or state overrides). Pendle's hosted API (api-v2.pendle.finance) does NOT automatically surface permissionless markets: on-chain market creation is open to all, but visibility on Pendle's app/API is curated — after a community pool deploys via the listing portal (listing.pendle.finance) the Pendle team manually sets up backend metadata/pricing and whitelists the pool; market objects carry isWhitelistedPro/isWhitelistedSimple/isWhitelistedLimitOrder/isActive flags. So OpenPendle's paste-an-address loading must read everything on-chain (market.readTokens(), SY.getTokensIn/Out, RouterStatic quotes) — which is exactly what the contracts support. The TypeScript SDK (pendle-sdk-core-v2-public) was archived Feb 19, 2024; the maintained integration path is the hosted-SDK REST API (POST /v3/sdk/{chainId}/convert, GET v2 deprecated), which is rate-limited (CU system) and backend-dependent — unsuitable for OpenPendle's goals. Best open-source references: pendle-examples-public (active, Solidity + TS demos incl. hosted-sdk and limit-order demos), Pendle-Common-Pool-Deploy, and Pendle-SY-Public (150+ SY implementations). Zapping: the router natively handles any token in the SY's own getTokensIn()/getTokensOut() list (set TokenInput.tokenIn == tokenMintSy, SwapData.swapType = NONE); arbitrary-token zaps require external aggregator calldata (SwapType enum: KYBERSWAP, ODOS, ETH_WETH, OKX, ONE_INCH, PARASWAP) passed via TokenInput.swapData and executed through the PendleSwap contract — a third-party-API dependency, not a Pendle-backend one. A realistic backend-free v1: restrict zap tokens to SY tokensIn/tokensOut, optionally add client-side KyberSwap API calls later. Limit orders are irrelevant for a community-pool UI: fills require Pendle's off-chain orderbook (limit-order API) and per-market whitelisting; passing an empty LimitOrderData routes 100% through the AMM (verified in ActionBase._isEmptyLimit), so OpenPendle should always pass empty limits. Protocol fees (netSyFee) accrue inside the market/SY contracts regardless of which frontend calls the router, so Pendle treasury flows are preserved automatically.

## Pendle protocol fees: where they are set, enforced, and how a third-party UI (OpenPendle) interacts with them

**Summary:** All Pendle protocol fees are enforced entirely inside the core contracts, so any third-party UI automatically respects them; there is no frontend fee or on-chain referral system in Pendle's official app to bypass. (1) Swap fee: each market stores an immutable lnFeeRateRoot chosen by the pool deployer at createNewMarket (capped at ln(1.05)); at every swap the market calls factory.getMarketConfig(market, router) to fetch the treasury address and reserveFeePercent (currently 80 on Arbitrum), computes the fee in MarketMathCore.calcTrade, and transfers netSyToReserve (80% of the fee, in SY) directly to Pendle's treasury (0xCbcb48e2...b05e6 on Arbitrum) in the same transaction; the remaining 20% stays in the pool for LPs. (2) YT fee: interestFeeRate and rewardFeeRate live on the YieldContractFactory (governance-set, capped at 20%; currently 5% = 5e16 on factories v5/v6, 3% on older v1/v3/v4 on Arbitrum) and are deducted in-contract at redemption (InterestManagerYT._doTransferOutInterest and PendleYieldToken reward logic), sent to the same treasury; post-expiry, ALL yield/rewards on unredeemed YT go to the treasury via redeemInterestAndRewardsPostExpiryForTreasury(). (3) One nuance: the factory owner can set per-router discounted fees (setOverriddenFee); Pendle's PendleFeeSetter dynamically sets a much lower swap fee for Pendle's canonical Router 0x8888...F946 (verified live: base fee ~1.27% ln-rate vs ~0.12% overridden for the official router on a V6 Arbitrum market). Therefore OpenPendle should route swaps through Pendle's canonical Router to give users the same (cheaper) fee as the official app — fee override is keyed to the router contract, not the frontend. Nothing special is needed to "keep fees flowing" to Pendle: any interaction path (official router, custom router, or direct market calls) pays protocol fees to the treasury; a custom router would actually pay MORE (the non-overridden base fee). The only fee OpenPendle must not tamper with is nothing — it cannot: treasury and reserveFeePercent are factory-level and governance-controlled, not settable by market deployers or callers.

**Key facts:**
- Swap fee parameter lnFeeRateRoot is an immutable per-market value (uint80) chosen by the pool deployer in PendleMarketFactory.createNewMarket(address PT, int256 scalarRoot, int256 initialAnchor, uint80 lnFeeRateRoot); capped by maxLnFeeRateRoot = ln(1.05) = 48790164169432003 (1e18 fixed-point). Market creation is permissionless ('Anyone is allowed to create a market on their own' - factory source).
- Fee tier semantics: lnFeeRateRoot = ln(1 + feeTier); effective trading fee scales with time to expiry: feeRate = exp(lnFeeRateRoot * timeToExpiry / 365 days) (MarketMathCore._getExchangeRateFromImpliedRate, used as comp.feeRate in calcTrade). Docs formula: Trading Fee = (Fee Tier / 365) * Days to Maturity. MarketDeployLib.calcFee(fee) = ln(1+fee) confirms UI fee-tier mapping.
- Treasury share of swap fee: MarketMathCore.calcTrade computes netAssetToReserve = fee * reserveFeePercent / 100; the market transfers it in SY directly to treasury inside each swap: IERC20(SY).safeTransfer(market.treasury, netSyToReserve) (PendleMarketV7.swapExactPtForSy / swapSyForExactPt; same pattern in V3). The remaining (100 - reserveFeePercent)% of the fee stays in the pool and accrues to LPs.
- treasury and reserveFeePercent live on the MARKET FACTORY, not the market; set by factory owner (governance) via setTreasuryAndFeeReserve(address,uint8), max reserveFeePercent = 100. Markets fetch them at every swap via readState(router) -> IPMarketFactory(factory).getMarketConfig(address market, address router) returns (treasury, overriddenFee, reserveFeePercent). Verified live on Arbitrum: reserveFeePercent = 80 on marketFactory V3-V6; treasury = 0xCbcb48e22622a3778b6F14C2f5d258Ba026b05e6 everywhere (matches deployments/42161-core.json 'network.treasury').
- Per-router fee override: mapping overriddenFee[router][market] (uint80), set only by factory owner via setOverriddenFee(router, market, newFee), must be < market's base fee (0 allowed). PendleMarketV7.readState: market.lnFeeRateRoot = overriddenFee == 0 ? lnFeeRateRoot : overriddenFee. The 'router' is msg.sender of the market call. PendleFeeSetter.sol (offchain-helpers) dynamically recomputes discounted fees for Pendle's canonical Router based on each market's implied rate ('dynamic fee rebalancing' from the Pendle 2025: Zenith roadmap) via governanceProxy.
- Live proof of router discount (Arbitrum, July 2026): USDai market 0xa8a0dea40174cfc30fea9e3a77f182ab33f46e25 (factory = marketFactoryV6 0x49F2f7002669E0e4425Fa0203975625Ab4af3143): base getNonOverrideLnFeeRateRoot() = 12718772407774688 (~1.28% tier); getMarketConfig(market, officialRouter 0x888888888889758F76e7103c6CbF23ABbF58F946) returns overriddenFee = 1232964221948952 (~0.12%); for any other caller overriddenFee = 0 (pays full base fee). reserveFeePercent = 80 applies to whichever fee is charged, so treasury is paid in all paths.
- YT interest/reward fee: interestFeeRate and rewardFeeRate (uint128, 1e18 fixed-point) live on PendleYieldContractFactory; governance-settable via setInterestFeeRate/setRewardFeeRate, hard caps maxInterestFeeRate = maxRewardFeeRate = 2e17 (20%). Enforced at redemption inside the YT contract: InterestManagerYT._doTransferOutInterest reads interestFeeRate() and treasury() from the factory, deducts feeAmount = interestPreFee.mulDown(feeRate), transfers fee in SY to treasury (emits CollectInterestFee); reward fee identically in PendleYieldToken._doTransferOutRewards (emits CollectRewardFee). No UI can bypass this.
- Current live YT fee values on Arbitrum (verified via eth_call, July 2026): yieldContractFactory v1 (0x28dE02Ac...), v3 (0xEb38531d...), v4 (0xc7F8F9F1...): interestFeeRate = rewardFeeRate = 3e16 (3%); v5 (0xFF29e023...) and v6 (0xBA814Bf6...): 5e16 (5%) - matching docs 'Pendle collects a 5% fee from all yield accrued (including points) by all YT in existence'. All factories' treasury = 0xCbcb48e2...b05e6. expiryDivisor = 86400 (expiries must be 00:00 UTC-aligned).
- Expired markets: swaps and mint (add liquidity) are blocked post-expiry (notExpired modifier; MarketExpired error), but burn/removeLiquidity still works. Post-expiry, all further interest+rewards accruing to unredeemed YT positions belong to the treasury: anyone can call YT.redeemInterestAndRewardsPostExpiryForTreasury(); '_redeemExternalReward(); // do a final redeem. All the future reward income will belong to the treasury' (PendleYieldToken). PT redemption post-expiry (redeemPY) carries no protocol fee. Docs: 'All yield and points generated by these unredeemed, matured positions are automatically redirected to the Pendle treasury fee wallet.'
- Mint/redeem PT+YT (mintPyFromSy/redeemPyToSy) and SY wrap/unwrap (deposit/redeem) carry NO Pendle protocol fee; protocol fees exist only on (a) market swaps, (b) YT yield/reward accrual, (c) post-expiry unredeemed YT yield, (d) limit order fills.
- Limit orders (separate system): PendleLimitRouter (0x000000000000c9B3E2C3Ec88B1B4c0cD853f4321 on Arbitrum) charges a fill fee in SY, transferred to a governance-set feeRecipient; per-YT lnFeeRateRoot set by governance via setLnFeeRateRoots; getLnFeeRateRoot(YT) reverts with 'LOP: fee not set' for YTs governance has not configured - so limit orders simply will not work for arbitrary community-pool YTs until Pendle sets their fee (OpenPendle should degrade gracefully).
- No frontend fee / no referral fee: Pendle's official app adds no fee on top of protocol fees - docs Fees page lists only YT 5% + swap fees and 'contains no information about interface/frontend fees, referral fees'; the Router contracts contain no frontend-fee or referral parameter. Fee distribution downstream of the treasury (currently docs: 20% of swap fees to LPs on-chain; of the treasury share: 80% PENDLE buyback, 10% Protocol Treasury, 10% Protocol Operations) is off-chain policy and irrelevant to UI integration.
- What OpenPendle should DO: (1) route swaps/zaps through Pendle's canonical RouterV4 0x888888888889758F76e7103c6CbF23ABbF58F946 so users pay the same dynamically-discounted fee as app.pendle.finance (override is keyed to router address, not frontend); (2) deploy markets through the standard factories (marketFactoryV5 0xd29e76c6F15ada0150D10A1D3f45aCCD2098283B / V6 0x49F2f7002669E0e4425Fa0203975625Ab4af3143 on Arbitrum) and yield-contract factories so treasury/reserveFeePercent/YT fees apply automatically; (3) read fee config via market.readState(routerAddr) and factory.getMarketConfig(market, router) rather than hardcoding. What to AVOID: nothing fee-related can be bypassed even deliberately - deployers cannot set treasury or reserveFeePercent, and lnFeeRateRoot=0 markets are effectively prevented because swaps revert with MarketZeroNetLPFee when index.syToAsset(netSyFee - netSyToReserve) == 0 (with reserveFeePercent=100 all swaps would revert; with fee=0 all swaps revert).
- Repo state (github.com/pendle-finance/pendle-core-v2-public main, July 2026): current market code is PendleMarketV7.sol + PendleMarketFactoryV7Upg.sol (VERSION=7) and PendleYieldContractFactoryUpg.sol (VERSION=6); Arbitrum deployments JSON lists factories up to V6 (marketFactoryV6 0x49F2..., yieldContractFactoryV6 0xBA81...) - live markets checked use V6. Factory getters: getMarketConfig(address,address) selector 0x5c098c11, reserveFeePercent() 0x1feec244, treasury() 0x61d027b3, interestFeeRate() 0xdd86fea1, rewardFeeRate() 0xf5de2d1f, getNonOverrideLnFeeRateRoot() 0xe4f8b2e9.

## Pendle V2 post-expiry behavior of PT/YT/markets and required UI handling for OpenPendle (Arbitrum)

**Summary:** Verified against pendle-core-v2-public (main branch, July 2026) and docs.pendle.finance. At expiry, everything is handled lazily and permissionlessly on-chain — no keeper is required. The YT contract's `updateData` modifier auto-calls `_setPostExpiryData()` on the first post-expiry interaction, snapshotting `firstPYIndex` (current PY index) and per-reward-token `firstRewardIndex`. After expiry: (1) PT redeems without YT — the router's `redeemPyToSy/redeemPyToToken` helper `_redeemPyToSy` only transfers/burns YT `if (!IPYieldToken(YT).isExpired())`; the user receives `SYUtils.assetToSy(pyIndexCurrent, amountPY)` SY, i.e. exactly 1:1 accounting-asset value at redemption time, and the interest earned by that SY between the post-expiry snapshot and redemption (`assetToSy(firstPYIndex, amountPY) - syToUser`) accrues to `postExpiry.totalSyInterestForTreasury` and is swept to Pendle's treasury via `redeemInterestAndRewardsPostExpiryForTreasury()`. (2) YT stops accruing: interest is capped at `firstPYIndex` and reward indexes are frozen at `firstRewardIndex`, so YT is worthless going forward, but residual pre-expiry interest/rewards remain claimable forever via `redeemDueInterestAndRewards(user, redeemInterest, redeemRewards)` (also batched via router `redeemDueInterestAndRewards(user, sys, yts, markets)`). Minting PT+YT reverts post-expiry (`notExpired` → `Errors.YCExpired()`). (3) The AMM: `mint` (add liquidity) and both swap entrypoints (`swapExactPtForSy`, `swapSyForExactPt`) carry the `notExpired` modifier / revert `Errors.MarketExpired()` in `MarketMathCore.executeTradeCore` and `addLiquidityCore`; `burn` (remove liquidity) has NO expiry check and always works, returning SY + PT pro-rata; LP `redeemRewards(user)` also has no expiry restriction. Router single-sided exits handle expiry explicitly: `_removeLiquiditySingleSy` burns LP then, if expired, redeems the PT leg via `YT.redeemPY` instead of swapping, so `removeLiquiditySingleSy/removeLiquiditySingleToken` and the one-click `exitPostExpToSy/exitPostExpToToken(receiver, market, netPtIn, netLpIn, ...)` all work post-expiry with zero price impact. A UI must therefore: disable swap PT/YT, mint PT+YT, add-liquidity and zap-in when `market.isExpired()`; surface "Redeem PT" (PT-only, no YT input), "Claim YT interest/rewards", and "Remove liquidity / exit" (dual or single-sided via post-expiry path); treat YT balance as $0; and price PT at 1 accounting asset (convert to SY/underlying via pyIndexCurrent = max(SY.exchangeRate(), pyIndexStored), non-decreasing even if SY rate drops). Pendle's own app tags such markets "[Matured]" with a `/matured` route, removes them from active listings, and offers only redeem/claim/withdraw ("will automatically perform Redemption > Swap (if needed)"). SY wrap/unwrap is unaffected by expiry (SY has no expiry). Treasury fee flows are preserved automatically by using the canonical YT/market contracts.

**Key facts:**
- PT redemption post-expiry needs NO YT: router helper _redeemPyToSy (contracts/router/base/ActionBase.sol) does `bool needToBurnYt = (!IPYieldToken(YT).isExpired()); if (needToBurnYt) _transferFrom(IERC20(YT), msg.sender, YT, netPyIn);` then calls `IPYieldToken(YT).redeemPY(receiver)`. Pre-expiry the same functions require equal amounts of PT AND YT.
- Router (PendleRouterV4, 0x888888888889758F76e7103c6CbF23ABbF58F946 on Arbitrum and all chains) signatures: `redeemPyToSy(address receiver, address YT, uint256 netPyIn, uint256 minSyOut)` and `redeemPyToToken(address receiver, address YT, uint256 netPyIn, TokenOutput calldata output) returns (uint256 netTokenOut, uint256 netSyInterm)` — note the second arg is the YT address, not PT or market.
- Low-level path: transfer PT (and YT if pre-expiry) directly to the YT contract, then call `redeemPY(address receiver)` (or `redeemPYMulti`); NatSpec: 'PT/YT must be transferred to this contract prior to calling'. Inside, PT is burned via `IPPrincipalToken(PT).burnByYT(...)` and the YT `_burn` is wrapped in `if (!isExpired())`.
- 1:1 semantics: post-expiry the PT holder receives `syToUser = SYUtils.assetToSy(indexCurrent, amountPY)` — i.e. exactly amountPY of accounting asset valued at redemption time. The gap vs the expiry snapshot, `SYUtils.assetToSy(postExpiry.firstPYIndex, amountPY) - syToUser`, accrues to `postExpiry.totalSyInterestForTreasury` (Pendle treasury), NOT the user. So a UI should quote expired PT at 1 accounting asset and convert to SY/underlying with the current index.
- pyIndexCurrent = `PMath.max(IStandardizedYield(SY).exchangeRate(), _pyIndexStored)` — monotonically non-decreasing, keeps updating after expiry. Docs: 'Upon maturity, PT can be redeemed at 1:1 for the accounting asset' (e.g. 1 PT-ezETH redeems 1 ETH worth of ezETH, not 1 ezETH).
- No keeper/settlement step: YT's `updateData` modifier runs `if (isExpired()) _setPostExpiryData();` on every state-changing call, so the first post-expiry interaction (a redeem, claim, or explicit permissionless `setPostExpiryData()`) initializes PostExpiryData {firstPYIndex, firstRewardIndex[], userRewardOwed[], totalSyInterestForTreasury}. A UI never needs to settle anything, but MAY call setPostExpiryData() to pin the snapshot early.
- YT after expiry: interest accrual is capped at firstPYIndex and reward accrual frozen at firstRewardIndex (`if (isExpired()) indexes[i] = postExpiry.firstRewardIndex[tokens[i]];` in _updateRewardIndex). YT market value is 0 (docs: 'YT value goes to 0 upon maturity'). Residual interest/rewards accrued before the snapshot remain claimable indefinitely via YT.redeemDueInterestAndRewards(address user, bool redeemInterest, bool redeemRewards) — callable by anyone for any user, proceeds go to the user. Router batch: `redeemDueInterestAndRewards(address user, address[] sys, address[] yts, address[] markets)`.
- Post-expiry SY interest and rewards accrued by the YT contract go to Pendle's treasury via `redeemInterestAndRewardsPostExpiryForTreasury()` (reverts YCNotExpired if pre-expiry) — treasury fee flow is automatic when using canonical contracts; OpenPendle needs no extra work to preserve it.
- Minting PT+YT post-expiry is impossible: `mintPY`/`mintPYMulti` have `notExpired` modifier reverting `Errors.YCExpired()`. Router mintPyFromToken/mintPyFromSy therefore revert too — UI must disable mint on expired markets.
- AMM post-expiry: swaps hard-revert at contract level — `MarketMathCore.executeTradeCore` and `addLiquidityCore` both contain `if (MiniHelpers.isExpired(market.expiry, blockTime)) revert Errors.MarketExpired();`, and PendleMarket `mint`, `swapExactPtForSy`, `swapSyForExactPt` carry a `notExpired` modifier. So swap PT, swap YT (flash-swap based), add liquidity, and all zap-ins are dead after expiry.
- Removing liquidity always works: `burn(address receiverSy, address receiverPt, uint256 netLpToBurn)` has NO expiry check (removeLiquidityCore has no expiry guard) and pays out SY + PT pro-rata. LP `redeemRewards(address user)` also has no expiry restriction on current market versions.
- Single-sided LP exit post-expiry (no swap, no price impact): router `_removeLiquiditySingleSy` does `if expired: IPMarket(market).burn(...); netSyOut = syFromBurn + YT.redeemPY(receiver)` else swaps PT→SY. Exposed as removeLiquiditySingleSy / removeLiquiditySingleToken (netSyFee = 0 in expired path) and as one-click `exitPostExpToToken(address receiver, address market, uint256 netPtIn, uint256 netLpIn, TokenOutput output)` / `exitPostExpToSy(receiver, market, netPtIn, netLpIn, minSyOut)` which also fold in loose PT. Pre-expiry counterparts: exitPreExpToToken/exitPreExpToSy (take netPtIn, netYtIn, netLpIn, LimitOrderData).
- Pendle's own UI: matured markets get a '[Matured]' title prefix and a distinct `/trade/markets/<address>/matured` route, are removed from the active market list, trading/LP-add actions are hidden/disabled, and the dashboard offers manual 'Redeem' (docs: the app 'will automatically perform Redemption > Swap (if needed)') plus yield/reward claiming. Docs stress redemption is manual — unredeemed expired PT just sits idle (opportunity cost, no loss).
- Edge — long-expired markets: nothing decays; redeemPY, burn, redeemDueInterestAndRewards, redeemRewards all remain callable years later; the PostExpiryData snapshot is simply taken at the first interaction whenever that happens. Subtlety: YT technically keeps accruing interest/rewards until that first post-expiry poke (firstPYIndex is set at poke time, not the expiry timestamp), so 'YT stops at maturity' is approximate at the contract level.
- Edge — SY exchangeRate still moving after expiry: harmless to PT holders in accounting-asset terms (they always get 1:1 at current index; extra yield goes to treasury). Because pyIndex is a max(), a falling SY exchangeRate does not lower the index — but if the SY's underlying genuinely depegs, the SY received per PT is fixed by the (higher) stored index and its real value can be below 1 accounting asset; UI should value redemption output via SY.previewRedeem/exchangeRate rather than assuming par.
- Edge — old markets' redeemRewards can revert: docs Troubleshooting notes 'An older implementation caused reward calculations to be incorrect after pool expiry. A reward redeem proxy contract was deployed as a fix. To redeem rewards from affected markets, call the proxy instead of the market contract directly.' Affects legacy markets only; new community pools (MarketFactory V5/V6, PendleMarketV7 code) are not affected.
- SY wrap/unwrap (SY.deposit / SY.redeem, router mintSyFromToken/redeemSyToToken) has no expiry concept and keeps working forever — expired-market UI can still offer unwrap of SY received from LP burn or PT redemption.
- UI state machine per market: `isExpired = market.isExpired()` (== `MiniHelpers.isCurrentlyExpired(expiry)`, i.e. block.timestamp >= expiry, readable via market.expiry() / readTokens+readState). If expired: enable {redeem PT (PT-only input), claim YT interest+rewards, claim LP rewards, remove liquidity dual (SY+PT then chain PT redemption) or single (router expired path), unwrap SY}; disable {swap PT, swap YT, mint PT+YT, add liquidity, zap in}; show YT as $0 with claimable residuals; show PT at 1 accounting asset.

## Backend-free reading/computation of Pendle market data for an arbitrary market (PLP) address on Arbitrum, plus oracle usage, indexer landscape, prior art, and recommended 2026 frontend stack

**Summary:** Everything OpenPendle must display can be derived from on-chain reads given only the market (PLP) address, with no Pendle backend. Flow: market.readTokens() returns (SY, PT, YT); SY.assetInfo()/exchangeRate()/getTokensIn()/getTokensOut() describe the underlying; market.readState(address(0)) returns MarketState (totalPt, totalSy, totalLp, treasury, scalarRoot, expiry, lnFeeRateRoot, reserveFeePercent, lastLnImpliedRate) — Pendle's own PendleLpOracleLib calls readState(address(0)), so the zero router is safe for reads. Implied APY = e^(lastLnImpliedRate/1e18) − 1 (rate is annualized over exactly 365 days per MarketMathCore.IMPLIED_RATE_TIME); spot PT-in-asset = 1/exp(lnRate·timeToExpiry/(365d·1e18)) and is exactly 1.0 post-expiry; YT-in-asset = 1 − PT-in-asset (this identity is how the oracle lib computes it), YT leverage ≈ 1/ytPrice. TWAP prices come from the singleton PendlePYLpOracle at 0x5542be50420E88dd7D5B4a3D488FA6ED82F6DAc2 (same address on all chains; key "pyYtLpOracle" in deployments/42161-core.json) via getPtToSyRate/getYtToSyRate/getLpToSyRate(market, duration) — but community markets deploy with observation cardinality 1, so the UI must run the getOracleState → increaseObservationsCardinalityNext(cardinalityRequired) → wait-duration dance (Arbitrum blockCycleNumerator=1000, so a 900s TWAP needs cardinality ~901), or fall back to duration=0 spot with a manipulation warning. Pool TVL in underlying-asset terms = (totalSy·SY.exchangeRate() + totalPt·ptToAssetRate)/1e18; USD display requires an external price source since no on-chain USD feed exists for arbitrary assets. User positions: plain balanceOf for PT/YT/LP/SY; claimable interest/rewards via RouterStatic (0xAdB09F65bd90d19e3148D9ccb693F3161C6DB3E8, ERC-2535 diamond) IPActionInfoStatic facet — getUserSYInfo/getUserPYInfo/getUserMarketInfo are state-mutating but callable with eth_call — or raw via YT.userInterest(user), YT.userReward(token,user), SY.accruedRewards(user), and eth_call-simulating market.redeemRewards(user). Historical underlying APY (7-day syIndex growth per Pendle's documented formula) needs eth_call at old block numbers, i.e. an archive-capable RPC. Indexers: Pendle runs 40+ private subgraphs on Ormi 0xgraph (no public endpoints); the public REST API api-v2.pendle.finance/core covers Arbitrum but is rate-limited (100 CU/min/IP) and defeats the backend-free goal; no verified public Arbitrum subgraph exists. Prior art: no meaningful third-party open-source Pendle frontend exists; study pendle-examples-public (Foundry samples OracleSample.sol/RouterSample.sol/RouterStaticSample.sol + hosted-SDK/API demos) and the archived pendle-sdk-core-v2-public TS SDK (archived Mar 2026 in favor of the hosted API — the last full open-source client-side implementation of Pendle reads/writes). Stack (verified July 2026): viem 2.54.x + wagmi (2.19.5 if you want RainbowKit 2.2.11, which still pins wagmi ^2.9.0; or wagmi 3.6.x + Reown AppKit 1.8.x whose peer range ≥2.19.5 admits v3) + TanStack Query 5.101.x + Vite 8.1.x SPA (static/IPFS-hostable, fits backend-free ethos better than Next 16) + viem's built-in Multicall3 batching + @wagmi/cli codegen from the @pendle/core-v2 npm package (v6.7.1).

**Key facts:**
- Market discovery from PLP address alone: readTokens() external view returns (IStandardizedYield _SY, IPPrincipalToken _PT, IPYieldToken _YT); expiry() returns uint256; isExpired() returns bool; factory() returns address; getRewardTokens() returns address[] (verified in IPMarket.sol / PendleMarketV7.sol on main branch of pendle-core-v2-public).
- readState(address router) returns MarketState { int256 totalPt; int256 totalSy; int256 totalLp; address treasury; int256 scalarRoot; uint256 expiry; uint256 lnFeeRateRoot; uint256 reserveFeePercent; uint256 lastLnImpliedRate; } — router param only affects fee fields via IPMarketFactory.getMarketConfig(market, router); Pendle's own PendleLpOracleLib calls market.readState(address(0)), so address(0) is safe for display reads.
- Cheaper single-slot alternative: _storage() external view returns (int128 totalPt, int128 totalSy, uint96 lastLnImpliedRate, uint16 observationIndex, uint16 observationCardinality, uint16 observationCardinalityNext).
- Underlying asset info: SY.assetInfo() returns (AssetType assetType, address assetAddress, uint8 assetDecimals) (0=TOKEN, 1=LIQUIDITY; doc comment also mentions 2 for bridged yield tokens whose underlying doesn't exist on-chain, e.g. wstETH on Arbitrum — decode as uint8, don't assume 2 members). Also SY.exchangeRate() (1e18-scaled SY→asset index), SY.yieldToken(), SY.getTokensIn()/getTokensOut() (zap-in/out token lists), SY.previewDeposit(tokenIn, amt) / SY.previewRedeem(tokenOut, amt) for wrap/unwrap quoting, SY.decimals().
- Implied APY (client-side, zero RPC beyond readState): impliedApy = e^(lastLnImpliedRate/1e18) − 1. Verified in Pendle's APY Calculation doc and MarketMathCore.sol where IMPLIED_RATE_TIME = 365 * 86400 and exchangeRate = exp(lnImpliedRate·timeToExpiry/IMPLIED_RATE_TIME) — the rate is annualized ln(1+apy) scaled 1e18.
- Spot PT price in asset = 1e18·1e18 / exp(lnImpliedRate·timeToExpiry/(365d)) (PendlePYOracleLib.getPtToAssetRateRaw); returns exactly PMath.ONE (1e18) once expiry <= block.timestamp — this is the canonical expired-market display rule. PT price in SY = ptToAssetRate / pyIndex (lib divides by max(syIndex, pyIndexStored) with an insolvency guard: if SY.exchangeRate() < pyIndex, multiply by syIndex/pyIndex).
- YT pricing: getYtToAssetRateRaw = 1e18 − getPtToAssetRateRaw (exact identity, since PT+YT mint/redeem to 1 asset-unit of SY). YT leverage for display ≈ 1e18/ytToAssetRate. Post-expiry YT trades at 0 and swaps revert (MarketMathCore reverts MarketExpired on any swap when expired) — expired UI must switch to redeem-only flows.
- Client-side TWAP without the oracle contract: market.observe(uint32[] secondsAgos) returns uint216[] lnImpliedRateCumulative; TWAP lnRate = (cum[now] − cum[now−duration]) / duration — exactly what PendlePYOracleLib.getMarketLnImpliedRate does (duration 0 short-circuits to _storage lastLnImpliedRate).
- PendlePYLpOracle (current singleton, same address on every supported chain): 0x5542be50420E88dd7D5B4a3D488FA6ED82F6DAc2 — matches both docs (Contracts/Oracle/PYLpOracle.md: find under "pendlePYLpOracle"/"pyYtLpOracle" in deployments) and deployments/42161-core.json. - Oracle initialization is mandatory for community pools: markets deploy with observation cardinality 1. Sequence: getOracleState → if increaseCardinalityRequired, anyone calls IPMarket(market).increaseObservationsCardinalityNext(cardinalityRequired) → wait `duration` seconds until oldestObservationSatisfied. On Arbitrum blockCycleNumerator = 1000, so cardinalityRequired ≈ duration+1 (900s → ~901 slots; docs table says 900 for Arbitrum vs 85 on Ethereum). Solidity libs PendlePYOracleLib/PendleLpOracleLib can be inlined instead of the contract (~4k gas cheaper) — for on-chain integrators, not needed by a JS frontend.

- 23 of 24 CreateNewMarket tx receipts were confirmed (one call was dropped by RPC rate limiting mid-batch); the aggregate conclusion (commonDeploy dominance) is robust but the one unchecked receipt should be re-verified if exactness matters.
